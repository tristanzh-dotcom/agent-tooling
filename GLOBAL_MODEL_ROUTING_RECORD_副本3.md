# Global Model Routing Record

Last updated: 2026-06-18

Scope: all Codex-managed projects under `/Users/tristanzh/agent`.

This file is the shared startup record for model routing, privacy boundaries, and first-principles usage rules. Every project should read this file before making assumptions about model responsibility.

## First-Principles Rule

Facts are not generated by language models.

For sales, rankings, YTD totals, brand/model quantities, schedules, source URLs, and other factual values:
- use deterministic parsers, source-specific collectors, cached verified data, and validation rules;
- never accept an LLM output as the final factual source;
- if an LLM is used to parse or normalize text, treat its output as a candidate that must pass schema, range, period, and source checks;
- mark uncertain or structurally incomplete data with the existing `（待查）` convention.

LLMs may be used for:
- Chinese semantic understanding;
- Chinese wording and summarization;
- translation or text normalization;
- workflow assistance where the output is not itself a factual number.

## Required Usage Routine

When reading this file during project startup:
1. Identify the current project.
2. Match the project to the most relevant route rows in `Current Project Routes`.
3. State which responsibilities are deterministic-only and which may use LLMs.
4. State privacy impact before invoking any external provider.
5. If no route matches the current project, do not invent one; mark routing as `unregistered` and ask TZ whether to add a route.

## Unregistered Project Rule

If the current project is not listed in `Current Project Routes`:
- do not assume an external model provider;
- default to local/deterministic processing where possible;
- do not send project data to any external provider until TZ approves a route;
- if a new route is approved, update the project implementation/config, project tests, and this global model routing record together.

## Privacy Boundary Rule

Before invoking any route whose `Privacy Status` indicates data leaves the local machine or local deterministic/runtime boundary, explicitly state:
- provider;
- data category to be sent;
- purpose;
- expected output;
- whether raw files, images, retrieved chunks, or only text snippets are sent.

Routes marked `local only`, `local processing`, `local deterministic processing`, or `local runtime route` are treated as local unless the implementation sends data to an external provider.

Do not send raw files, image content, retrieved chunks, secrets, private notes, or project data to an external provider unless the route explicitly allows that data category or TZ approves it.

## Fact Confidence Labels

For factual outputs, use one of:
- `verified`: deterministic source + validation passed;
- `candidate`: parser, collector, or LLM output that has not yet passed source, schema, range, period, and consistency validation;
- `pending`: structurally incomplete or missing source; use `（待查）`;
- `not_applicable`: non-factual wording, summary, translation, or workflow text.

## Key Handling

- Do not store API keys in code.
- Do not print API keys, webhook tokens, app secrets, or access tokens in chat or logs.
- Project-local `.env` files may contain placeholders or runtime-loaded values, but source files should only reference environment variables.
- Reports should state key status as set/unset/placeholder/masked only.

## Update Rule

When a project model route is added, removed, renamed, changes provider, changes privacy status, or changes responsibility, update:
- the project implementation/config;
- the project tests;
- this global model routing record.

Route changes must not be bundled silently with unrelated feature work.
A route change requires an explicit note in the implementation summary or handover document.
