# 技能名称：收工交接文档生成 (Handover Skill)

## 触发条件
当用户要求以下任一意图时启用本 Skill：
- “交接”
- “收工”
- “今日收工”
- “handover”
- “closing”
- “end-of-day”
- “生成交接文档”
- 其他语义相近的收尾/交接/次日重启请求。

## 核心目标
本 Skill 的唯一目标是：在当前权威项目根目录内，生成一个高密度、客观真实、便于 TZ 人工审阅、修正和裁决，并作为次日启动上下文的交接文档。
*注：本 Skill 不是项目记忆更新器，不是 changelog 生成器，不是长期知识库维护器。*

## 全局硬规则

### 1. 权威项目根优先
禁止直接把当前工作目录（CWD）当成扫描锚点。在执行任何扫描、总结、写文件之前，必须先确认权威项目根。
判断优先级如下：
1. 当前目录及父目录中的 `AGENTS.md`（优先使用其声明的 Scope、明确路径或业务根目录）
2. 当前对话中用户明确指定的项目路径
3. 当前对话中明确指定的 handover 文件路径
4. 最近的 `HANDOVER_*_YYYYMMDD.md` 文件所在目录
如果候选项目根与 CWD 不一致，必须以权威项目根为准。如果仍无法唯一确认，必须停止执行并询问用户。

### 2. 禁止跨目录扫描
只能在已确认的权威项目根内扫描。严禁让 `rg`, `find`, `git`, `ls`, `tree` 或任何自定义脚本越界命中其他项目目录。
*示例：如果项目根位于 `/Users/tristanzh/agent/project_a`，则绝对不得扫描 `/Users/tristanzh/agent`。除非权威项目根本身被明确判定为 `/Users/tristanzh/agent` 整体，否则任何递归扫描命令不得越界。*

### 3. 单文件输出原则
本 Skill 只允许在项目根目录下创建一个文件：`HANDOVER_{主题标识}_YYYYMMDD.md`。
严禁擅自创建或更新任何其他历史交接、长期记忆或日志文件（如 `HANDOFF.md`, `PROJECT_MEMORY.md`, `CHANGELOG.md` 等）。

### 4. 禁止发明事实（反幻觉）
交接文档只能记录有确凿证据支持的信息。不得虚构已完成事项、已运行测试或用户未批准的结论。
如果证据不足，必须明确标记为 `【未确认】`。如果测试没有运行，必须写明原因并给出替代的手动验证路径。

### 5. 解耦双轨制证据优先级
在识别和提取信息时，必须严格按照以下两套不同的证据链进行归纳，不得交叉污染：

#### A. 文件变更证据优先级（客观代码状态）：
1. **第一优先级：Git 完整证据**（联合使用 `git status --short`, `git diff --stat`, `git diff --name-only`, `git diff --cached --name-only`，并配合 `git log --since="{时间边界}" --name-status --oneline` 以覆盖今日已提交的所有变更文件清单与历史细节。其中 `{时间边界}` 默认使用 `midnight`；若交接发生在 00:00–05:00，则改用 `24 hours ago` 或用户明确指定的上一工作日开始时间）
2. **第二优先级：文件修改时间**（`find <PROJECT_ROOT> -type f -mtime -1`，仅作为辅助线索，不能作为唯一证据）
3. **第三优先级：当前对话上下文**（结合对话中提及的修改动作进行辅助归纳）

#### B. 决策证据优先级（主观设计意图）：
1. **第一优先级：用户明确确认/授权的结论**
2. **第二优先级：当前对话上下文中的技术讨论与推演**
3. **第三优先级：代码变更推断**（仅作为最低权重的参考，严禁仅通过代码反推合并用户未明确表达的设计图景）

### 6. 排除非核心文件
以下文件不得默认列为“今日核心修改文件”：`node_modules/`, `.git/`, `dist/`, `build/`, `.next/`, `临时日志`, `自动生成预览图` 等编译缓存或系统文件。

### 7. 人类最高审查权与唯一真理源
生成的交接文档是供 TZ 人工审阅、修正和裁决的权威交接面板。
TZ 有权在关闭当前对话后，手动修改本地的 `HANDOVER_*.md` 文件。
次日新对话启动时，系统必须以本地文件中最终存在的实际文本为唯一真理源。
如果本地 handover 文件内容与模型记忆、历史对话印象或先前生成版本不一致，必须以本地文件为准。
不得用所谓“前一日模型记忆”反驳、覆盖或弱化 TZ 已经手动微调过的 handover 内容。

### 8. 同名文件处理规则
如果目标路径下 `HANDOVER_{主题标识}_YYYYMMDD.md` 已存在：
- **绝对禁止直接静默覆盖**，以防抹去 TZ 先前的手工微调内容。
- 必须先读取现有文件内容进行对齐，并**默认假设其中可能包含 TZ 手工修改**；除非用户明确授权覆盖，否则不得覆盖。
- 如果用户在对话中明确要求“更新/覆盖今天的交接文档”，方可覆盖。
- 必须主动询问 TZ：是覆盖现有文件，还是生成带序号的新版本文件。带序号格式严格递增为：`HANDOVER_{主题标识}_YYYYMMDD_v2.md`、`HANDOVER_{主题标识}_YYYYMMDD_v3.md` 等。在递增生成新序号版本时，也必须逐级检查目标文件是否已存在，绝对不得覆盖任何已有版本。

## 执行步骤

### Step 1：确认权威项目根
执行前必须严格根据【全局硬规则 1】完成路径判断。确认后，在内部将该目录锁定为唯一扫描范围。如果无法唯一确认，停止执行并询问用户。

### Step 2：收集今日变更证据
在权威项目根内收集以下客观信息：
- **Git 状态与变更历史（时序边界防御）**：全面获取仓库今日在工作区、暂存区和本地已 commit 记录中的变更文件清单及改动细节。
  *注意：若执行交接时处于凌晨阶段（00:00 - 05:00），系统时区或 `midnight` 关键字可能导致前一日白天 Commit 漏报。此时必须将 `--since` 时间显式前移扩展至涵盖前一日全天（如 `--since="24 hours ago"`），优先避免昨日核心变更在交接证据收集中被遗漏。*
- **今日对话决策**：依据【决策证据优先级】严格提取技术方案、放弃的路径及未解卡点。
- **测试与验证证据**：收集已运行的测试命令与测试结果。如果没有运行测试，不得写“验证通过”。

### Step 3：确定文件名与主题
根据当前对话的核心业务主题，提取内容并满足以下格式：
1. **简短英文主题标识**：用于文件名（小写英文，单词间用 `_` 连结，不含空格与中文）。格式为 `HANDOVER_{主题标识}_YYYYMMDD.md`。
2. **简短中文主题名称**：用于次日工作流锁定与对话自动重命名。

### Step 4：生成交接文档
在权威项目根目录生成 `HANDOVER_{主题标识}_YYYYMMDD.md`（或带有 `_v2` 等版本后缀）。

*注：下方模板在本 Skill 文件中使用四个反引号展示，是为了避免嵌套代码块闭合错误。实际生成的 `HANDOVER_*.md` 文件内容最外层绝对不得包含包裹整篇文档的反引号，只需保持文档内部各模块必要的正常 Markdown 代码块即可。*

## 失败处理机制
1. **无法确认项目根**：输出 `无法唯一确认权威项目根，因此未生成交接文档。请 TZ 指定本次 handover 应写入的项目路径。` 随后停止一切后续操作。
2. **无法写入文件**：输出 `交接文档生成失败。原因：{错误原因}`。若有部分写入的临时文件，必须清晰告知路径提示用户检查。

## 最小合格标准
一个合格的 handover 文档在交付时，必须确保 TZ 在次日开工前，即使不依赖任何模型历史记忆，也能一眼看清：当前项目根在哪、今天实际改了什么、测试跑了没有、当前最大的卡点是什么、以及明天第一步应该打开哪个文件的哪一行。如果证据不足，对应的模块必须大方写上“未确认”，绝不补编。
